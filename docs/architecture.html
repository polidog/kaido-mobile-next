プロジェクト概要

日本の五街道（東海道・中山道・甲州道中・日光道中・奥州道中）の宿場・名所を Google Maps 上に表示するモバイルアプリ群。 街道ごとに独立したアプリとしてストアに公開するが、コードベースは共通のモノレポで管理する。

データは kaido-web-next（Next.js 16 + Turso）の API から取得する。

旧アーキテクチャからの主な変更点 旧構成では kaido_core を Git URL / ローカルパスで参照し、5つの独立リポジトリに分散していた。 新構成では melos モノレポに統合し、Riverpod / go_router / freezed でモダナイズする。

アーキテクチャ方針

レイヤードアーキテクチャ

パッケージ単位でレイヤーを分離する。依存方向は 上位 → 下位 の一方向のみ。

apps/*
→
kaido_ui
→
kaido_data
→
kaido_api
→
kaido-web-next API
レイヤーパッケージ責務依存先
Presentation apps/* + kaido_ui 画面構成・ルーティング・ユーザー操作 kaido_data
Domain / Data kaido_data モデル定義・リポジトリ・状態管理（Riverpod） kaido_api
Infrastructure kaido_api HTTP 通信・JSON パース・エラーハンドリング なし（外部 API のみ）

各アプリの役割

各アプリ（apps/*）は設定とアセットの入れ物に過ぎない。 ビジネスロジックやUIは全て packages/ に集約し、アプリの main.dart は以下のみを担当する：

  • アプリ固有の設定（アプリ名、API コンテキスト、テーマカラー）を KaidoConfig として定義
  • Riverpod の ProviderScope に設定を注入
  • アプリ固有のアセットパスを指定
// apps/tokaido/lib/main.dart
void main() {
  runApp(
    ProviderScope(
      overrides: [
        kaidoConfigProvider.overrideWithValue(
          KaidoConfig(
            appName: '東海道五十三次',
            apiContext: 'tokaido',
            themeColor: Color(0xFFECB404),
            assetPrefix: 'assets',
          ),
        ),
      ],
      child: const KaidoApp(),
    ),
  );
}

@freezed
class Point with _$Point {
  const factory Point({
    required int id,
    required String title,
    required double lat,
    required double lng,
    required String description,
    required String category,
    String? image,
  }) = _Point;

  factory Point.fromJson(Map<String, dynamic> json) =>
      _$PointFromJson(json);

// repositories/point_repository.dart
class PointRepository {
  PointRepository(this._api, this._localStorage);

  final KaidoApiClient _api;
  final LocalDataSource _localStorage;

  Future<List<Point>> getPoints(String context) async {
    try {
      // API から取得を試みる
      final points = await _api.getPoints(context);
      await _localStorage.cachePoints(context, points);
      return points;
    } catch (_) {
      // フォールバック: ローカルキャッシュ or バンドル JSON
      return _localStorage.getPoints(context);
    }
  }
}

abstract class KaidoApiClient {

  Future<List<RoutePoint>> getRoutes(@Path('context') String context);

  @GET('/api/v1/maps/{context}/detours')

}

状態管理（Riverpod）

Riverpod のコード生成（@riverpod アノテーション）を使用する。 プロバイダは kaido_data/lib/providers/ に配置。

プロバイダ型責務
kaidoConfigProvider Provider<KaidoConfig> アプリ固有設定（名前・APIコンテキスト・テーマ色）
pointsProvider AsyncNotifier<List<Point>> ポイントデータの取得・キャッシュ管理
routesProvider AsyncNotifier<Set<Polyline>> ルート Polyline の取得・キャッシュ管理
detoursProvider AsyncNotifier<Set<Polyline>> 寄り道ルートの取得・キャッシュ管理
mapControllerProvider StateNotifier<MapState> GPS 追従・コンパスモード・カメラ位置
markerVisibilityProvider StateProvider<bool> マーカー表示/非表示の切替 
// providers/points_provider.dart
@riverpod
class Points extends _$Points {
  @override
  Future<List<Point>> build() async {
    final config = ref.watch(kaidoConfigProvider);
    final repository = ref.watch(pointRepositoryProvider);
    return repository.getPoints(config.apiContext);
  }

  Future<void> refresh() async {
    state = const AsyncLoading();
    state = await AsyncValue.guard(() async {
      final config = ref.read(kaidoConfigProvider);
      final repository = ref.read(pointRepositoryProvider);
      return repository.fetchAndCache(config.apiContext);
    });
  }
}

エラーハンドリング

kaido-web-next が採用している Result 型パターンに合わせ、 API 層でのエラーは sealed class で表現する。 

// kaido_api/lib/result.dart
sealed class ApiResult<T> {
  const ApiResult();
}

class ApiSuccess<T> extends ApiResult<T> {
  const ApiSuccess(this.data);
  final T data;
}

class ApiFailure<T> extends ApiResult<T> {
  const ApiFailure(this.error, [this.stackTrace]);
  final Object error;
  final StackTrace? stackTrace;
}

// 使用例
final result = await api.getPoints('tokaido');
switch (result) {
  case ApiSuccess(:final data):
    // ポイントを表示
  case ApiFailure(:final error):
    // フォールバック or エラー表示
}

地図マーカーのパフォーマンス最適化

クラスタリングは使用しない。カメラの表示範囲（VisibleRegion）内のマーカーのみを描画する。

  • ビューポートフィルタリング: onCameraIdle で getVisibleRegion() を取得し、範囲内のポイントだけを Marker として Set に反映
  • カスタムマーカー画像のキャッシュ: BitmapDescriptor をカテゴリごとに起動時一括生成し再利用

認証情報の管理

認証情報は .credentials/ に配置（.gitignore に追加済み）。 初回は ./release/release.sh setup で対話的にセットアップする。

ファイル用途取得元

google_play_service_account.json Google Play Console API 認証 Google Cloud Console → IAM → サービスアカウント
前提条件 iOS: Xcode + 有効な Apple Developer Program。 Android: gem install fastlane が必要。

マイグレーション計画

既存の5リポジトリから新モノレポへの段階的な移行手順。

Phase 1: モノレポ基盤構築

  • melos ワークスペースのセットアップ
  • kaido_api パッケージ作成（kaido-web-next の API クライアント）
  • kaido_data パッケージ作成（freezed モデル + リポジトリ）
  • kaido_ui パッケージ作成（kaido_core のコード移植 + Riverpod 化）

Phase 2: 東海道アプリで検証

  • apps/tokaido を新アーキテクチャで構築
  • 既存の東海道アプリと動作を比較検証
  • Google Maps のマーカーパフォーマンスを実機で確認

Phase 3: 全アプリ展開

  • 残り4アプリのプロジェクト作成（tokaido をテンプレートに）
  • アプリ固有のアセット・設定を配置
  • CI/CD パイプラインの構築

Phase 4: ストア移行

  • テストフライト / 内部テストで配信
  • 既存アプリのアップデートとして配信（バンドルID を維持）
